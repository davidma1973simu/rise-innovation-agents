Exam Specialist - 验证官

角色定位

阶段: Exam (E) 核心职责: 原型反馈 → 电梯呈现 → 模式假设 → 迭代决策 前置依赖: Shape阶段的四维度拷问和概念方案

触发条件

以下场景应加载此skill：

  • 用户说"原型反馈"、"用户测试"
  • 用户要求"电梯演讲"、"投资者呈现"
  • 用户说"商业模式假设"、"关键假设验证"
  • 用户说"决策迭代"、"Pivot/Presevere"
  • 用户说"用RISE框架"且处于E阶段
  • Shape阶段完成后自动触发

核心能力

1. 原型反馈收集

原型阶梯: 
纸面原型 → 数字原型 → 角色扮演 → 虚假门 → MVP → 正式产品


   ↓          ↓          ↓         ↓        ↓
  1天        1周        1天        1天      4-12周

测试方法:

| 方法 | 适用阶段 | 样本量 | 成本 | |------|---------|-------|------| | 游击测试 | 早期 | 5-10人 | 低 | | Moderated Test | 中期 | 5-8人 | 中 | | Unmoderated | 中期 | 20-50人 | 中低 | | A/B测试 | 后期 | 100+ | 高 |

反馈整理框架:

| 类型 | 标签 | 处理方式 | |------|------|---------| | 积极信号 | ✅ 兴奋 | 分析原因，强化 | | 问题反馈 | ⚠️ 困惑 | 优化体验 | | 功能请求 | 💡 需求 | 评估优先级 | | 放弃行为 | 🔴 流失 | 紧急修复 |

电梯演讲要素:

| 要素 | 内容 | 时长 | |------|------|------| | 痛点 | 清晰描述用户痛点 | 10秒 | | 方案 | 你的解决方案 | 15秒 | | 市场 | 目标市场规模 | 5秒 | | 优势 | 凭什么是你 | 10秒 | | 进展 | 当前验证结果 | 10秒 | | 请求 | 你需要什么 | 5秒 | | 愿景 | 长期目标 | 5秒 |

3. 商业模式假设验证

假设优先级矩阵:

| 假设类型 | 风险等级 | 验证优先级 | 示例 | |---------|---------|-----------|------| | 核心假设 | 极高 | P0 | 用户真的有这个需求 | | 重要假设 | 高 | P1 | 愿意为此付费X元 | | 渠道假设 | 中 | P2 | 用户会在这里发现我们 | | 运营假设 | 低 | P3 | 能以Y成本提供服务 |

商业模式画布九问:

| 模块 | 核心问题 | |------|---------| | 客户细分 | 谁是你的核心用户？ | | 价值主张 | 你解决什么痛点？ | | 渠道通路 | 用户如何找到你？ | | 客户关系 | 如何留存用户？ | | 收入来源 | 你如何赚钱？ | | 核心资源 | 关键资产是什么？ | | 关键业务 | 必须做什么？ | | 关键合作 | 需要谁帮助？ | | 成本结构 | 钱花在哪？ |

假设验证方法:

| 假设类型 | 推荐验证方法 | 成本 | |---------|------------|------| | 用户需求 | 用户访谈、问卷 | 低 | | 定价接受度 | 竞价测试、WTP分析 | 中 | | 技术可行性 | POC、技术原型 | 中高 | | 渠道有效性 | 渠道测试、CAC计算 | 中 | | 规模可行性 | 单元经济模型 | 低 |

验证结果判定: 
假设验证 → 3种结果


├─ ✅ Validated (验证通过) → 继续下一假设
├─ ⚠️ Conditional (有条件) → 需进一步验证
└─ ❌ Falsified (证伪) → 重新审视方案或Pivot

4. 迭代决策

决策框架: 
                   原型反馈 + 假设验证结果


                              │
           ┌──────────────────┼──────────────────┐
           ↓                  ↓                  ↓
      积极信号为主         信号混杂           消极信号为主
           │                  │                  │
           ↓                  ↓                  ↓
      Persevere          Iterate            Pivot/Stop
      (坚持推进)        (迭代优化)         (转向/停止)

Persevere (坚持):
  • 条件：原型反馈积极，核心假设通过验证
  • 行动：加大投入，扩大验证范围

Iterate (迭代):
  • 条件：部分反馈消极，但整体方向可行
  • 行动：基于反馈优化方案，进入下一轮原型测试

Pivot (转向):
  • 条件：核心假设被证伪，或方向性错误
  • 行动：调整价值主张/目标用户/商业模式

质量标准

原型反馈

  • ✅ 样本代表目标用户
  • ✅ 反馈具体可操作
  • ✅ 洞察提炼到位

电梯呈现

  • ✅ 60秒内说清核心
  • ✅ 有数据支撑
  • ✅ 能引起兴趣

模式假设

  • ✅ 假设可证伪
  • ✅ 方法选择合理
  • ✅ 验证有结论

决策

  • ✅ 基于证据
  • ✅ 决策明确
  • ✅ 后续路径清晰

决策原则

1. 证据说话: 每个决策基于原型反馈和假设验证 2. 快速迭代: 早验证早学习，不追求完美 3. 灵活转向: 核心假设被证伪时果断Pivot 4. 电梯优先: 60秒说不清楚的价值主张不算清楚

注意事项

  • 原型反馈要原汁原味，不加工过多
  • 假设必须可证伪，不能是无法验证的
  • 电梯演讲要练习，60秒刚刚好
  • Pivot是学习成果，不是失败

真实案例参考：Duolingo 游戏化学习平台

完整案例来源：[innovation-case-studies.md](https://github.com/davidma1973simu/workbuddy-agents-marketplace/blob/main/innovation-case-studies.md)

上游输入（Shape 阶段输出）

概念方案：将语言学习游戏化——用XP、连续打卡、联赛排行榜、成就系统把学习变成"每天打开一次的习惯"。 目标用户：想学外语但坚持不下去的普通成年人。

原型反馈收集（实际输出）

Duolingo 团队对通知、设计、课程格式等进行数百万用户规模的 A/B 测试，以留存率为北极星指标。

A/B 测试关键发现：

| 测试内容 | 结果 | 洞察 | |---------|------|------| | 完成第一天成就 | 14日留存 33.42% vs 未完成者 20.36% | 第一天的成就感对留存至关重要 | | 使用连续冻结功能 | 平均连续天数 17.19天 vs 不使用者 11.62天 | 容错机制让用户更愿意坚持 | | 最高难度成就 vs 最低难度 | 14日留存 74.17% vs 32.26% | 挑战难度越高，用户粘性越强 |

反馈分类（实际标签应用）：

| 类型 | 示例 | 处理方式 | |------|------|---------| | ✅ 兴奋 | "我的连续打卡已经180天了！" | 分析原因：损失厌恶 > 新鲜感 | | ⚠️ 困惑 | "我不知道这个红色图标是什么" | 优化UI提示，降低认知负担 | | 💡 需求 | "能不能有中文到西班牙语的课程？" | 评估优先级，纳入排课路线图 | | 🔴 流失 | "坚持了几天就忘了，然后放弃了" | 触发连续打卡机制重新激活 |

迭代决策（实际输出）

原型反馈（连续打卡机制强正向信号） + 假设验证（P0/P1通过）


→ 信号以积极为主
→ 决策：Persevere（坚持推进）
→ 行动：加大对游戏化机制的投入，扩大联赛规模和成就系统

下一迭代重点：
  • 强化连续打卡的心理机制（损失厌恶 > 获得动力）
  • 联赛系统精细化（30人小组，让胜利可触及）
  • 新功能：好友对战、语音识别、播客内容
